# 企业知识库 AI 助手权限设计说明

## 1. 为什么需要权限设计

企业知识库 AI 助手会接触企业内部文档，例如员工制度、客户资料、财务信息、部门流程等。

如果没有权限控制，AI 可能会把不该公开的信息回答给无权限用户，造成数据泄露。

因此，权限设计是企业 AI 项目上线前必须考虑的问题。

## 2. 用户角色设计

系统初步设计 4 类角色：

### 普通员工 Employee

权限：

- 可以向 AI 提问
- 可以查看自己有权限的答案
- 可以查看答案参考来源
- 不能上传文档
- 不能删除文档
- 不能查看全公司历史记录

### 部门主管 Manager

权限：

- 可以向 AI 提问
- 可以查看本部门知识库
- 可以上传本部门文档
- 可以删除本部门文档
- 可以查看本部门提问统计

### 知识库管理员 Admin

权限：

- 可以上传企业公共文档
- 可以删除过期或错误文档
- 可以维护知识库分类
- 可以查看用户反馈
- 可以管理文档可见范围

### 系统管理员 Super Admin

权限：

- 可以管理用户
- 可以设置角色权限
- 可以查看系统日志
- 可以配置模型和 API Key
- 可以管理全部文档和数据

## 3. 文档权限设计

每份文档需要保存权限信息，例如：

```text
file_name: employee_handbook.docx
department: HR
visibility: department
uploaded_by: admin_user
created_at: 2026-06-12 10:00:00

字段说明：
file_name：文档名称
department：所属部门
visibility：可见范围
uploaded_by：上传人
created_at：上传时间
visibility 可以分为：
public：全公司可见
department：仅部门可见
private：仅指定人员可见


4. 接口权限设计
POST /ask-doc
用户提问时，系统只能检索该用户有权限访问的文档。
例如：
普通员工只能查 public 文档和自己部门文档
部门主管可以查本部门文档
管理员可以查公共知识库
系统管理员可以查全部文档
POST /upload-doc
上传文档需要权限控制。
普通员工不能上传
部门主管可以上传本部门文档
知识库管理员可以上传企业公共文档
系统管理员可以上传所有类型文档
DELETE /documents/{file_name}
删除文档需要更高权限。
普通员工不能删除
部门主管只能删除本部门文档
知识库管理员可以删除公共文档
系统管理员可以删除所有文档
GET /history
历史记录也需要权限控制。
普通员工只能看自己的提问记录
部门主管可以看本部门统计
管理员可以查看反馈和知识库优化相关数据
系统管理员可以查看全部日志


5. 数据隔离设计
企业中不同部门的数据不能混在一起随便检索。
例如：
HR 文档不能随便给销售部门看
财务文档不能给普通员工看
管理层文件只能指定人员访问
所以 RAG 检索时，不能直接检索全部文档，而应该先根据用户权限过滤可访问文档，再在这些文档中检索相关片段。
流程：
用户提问
-> 判断用户身份和部门
-> 找出用户有权限访问的文档
-> 只在这些文档中做 RAG 检索
-> AI 基于允许访问的资料回答


6. 当前 Demo 与真实上线差距
当前 Demo 中：
没有用户登录
没有角色权限
上传入口只是前端折叠展示
所有文档默认都能被检索
history.json 保存本地历史记录
真实上线需要增加：
登录系统
用户角色表
文档权限表
部门字段
数据库保存用户、文档、历史记录
后端接口鉴权
操作日志和审计
